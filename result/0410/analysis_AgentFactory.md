# AgentFactory: A Self-Evolving Framework Through Executable Subagent Accumulation and Reuse

**来源**：arXiv:2603.18000v1 [cs.AI]  
**作者**：Zhang Zhang 等（北京大学、香港理工大学、北京人工智能研究院）  
**文件**：AgentFactory_Self-Evolving_Subagents_arXiv2603.18000.pdf

---

### 1. Task（任务定义）

- **核心问题**：如何让 LLM 智能体在完成任务的过程中持续积累并复用可靠的能力，而不是每次从零出发。
- **形式化表达**：  
  - 输入：用户任务序列 T1, T2, ..., Tn，以及初始工具集 ToolSet。  
  - 输出：随任务递增的可执行子智能体库 SubagentPool，满足对后续任务 Ti（i > k）的平均 token 消耗单调下降、任务成功率不降。  
  - 目标函数：在不引入人工干预的前提下，最大化 SubagentPool 的覆盖率与复用率，使得 Cost(Ti | SubagentPool) < Cost(Ti | from scratch)。

---

### 2. Challenge（传统挑战）

- **现有困境**：  
  - LangChain、AutoGPT 等框架**行为静态**，执行知识不持久，每次任务从零调用。  
  - Reflexion、Self-Refine 等「自进化」方法将成功经验存为**文本反思或 prompt**，对复杂多步任务无法可靠重现执行逻辑。  
  - AlphaEvolve、Darwin Gödel Machine 等代码进化方法**专为特定高复杂度科研或元推理**设计，不适用于通用日常工作流。  
  - 已有技能积累方法（Voyager 等）在跨平台可移植性和持续自主改进两方面存在局限。

### 3. Insight & Novelty（洞察与创新）

#### 3.1 Inspiration（灵感来源）

- 日常用户任务表面多样，但底层可拆分为**一组可复用的子任务**（如文件操作、日程安排、文献检索），这与软件工程中函数库累积的直觉一致。
- 代码相比自然语言具有**确定性语义与可执行性**，相同代码在任意 Python 环境中产生相同行为。
- Self-Refine 的「生成-反馈-修改」回路可从单次输出扩展到智能体代码本身。

#### 3.2 Insight（核心洞察）

- **策略层**：用「可执行子智能体代码」替代「自然语言经验」作为知识载体，实现任务执行模式的精确、可复用编码。  
  - 对应灵感：代码的确定性语义。  
- **架构层**：Meta-Agent 动态为子智能体**按需分配工具子集**，而非暴露全部工具集，压缩子智能体的搜索空间。  
  - 对应灵感：日常任务可拆分为有限子任务集。  
- **方法层**：在 Self-Evolve 阶段对已有子智能体基于**执行反馈做代码级修改**，而不是 prompt 级调整。  
  - 对应灵感：Self-Refine 回路可迁移至代码层。

#### 3.3 Novelty Type（创新类型）

方法层创新（将文本记忆升级为可执行代码库）+ 架构层创新（三阶段生命周期 + 工具子集动态分配）。

#### 3.4 Novelty Details（创新详述）

- 1️⃣ 【解决的问题是】：文本经验无法可靠重现复杂任务。  
  -> 【受哪个 Insight 启发】：代码具有确定性执行语义。  
  -> 【具体创新设计】：每次成功任务后，Meta-Agent 调用 create_subagent 将执行轨迹固化为纯 Python 脚本 + SKILL.md 文档，存入持久 SubagentPool；后续任务通过 list_saved_subagents 检索并直接复用。

- 2️⃣ 【解决的问题是】：子智能体初版代码对边界情况处理脆弱（如路径解析硬编码）。  
  -> 【受哪个 Insight 启发】：Self-Refine 生成-反馈-修改回路可以作用在代码层。  
  -> 【具体创新设计】：Self-Evolve 阶段，Meta-Agent 调用 run_subagent 得到失败反馈后，通过 modify_subagent 对代码做鲁棒性增强（错误处理、正则回退等），并重新验证，实现跨任务迭代改进（见 Fig.2 路径解析三版本演化案例）。

- 3️⃣ 【解决的问题是】：跨框架能力迁移困难。  
  -> 【受哪个 Insight 启发】：纯 Python 代码具有跨平台可移植性。  
  -> 【具体创新设计】：Deploy 阶段将子智能体导出为独立 Python 模块，外部系统（如 Claude Code）通过读取 SKILL.md 学会调用方式，无需代码级改造宿主框架（见 Fig.3 跨系统复用演示）。

### 4. Potential Flaw（潜在缺陷与拓展）

#### 4.1 情境局限与延伸

- SubagentPool 质量完全依赖**初始任务分布**：若早期任务集中于特定领域，池子会出现领域偏斜，跨域任务仍需从零构建。  
- 当任务数量爆炸时，list_saved_subagents 的**检索精度**成为瓶颈——若子智能体数量达到数百甚至数千，语义检索质量直接影响复用率。  
- 延伸方向：引入分层 / 聚类式子智能体索引，以及基于任务相似度的主动蒸馏机制。

#### 4.2 数据脆弱性

- **执行反馈噪声高**：若任务评估信号（判断子智能体「成功」的标准）不精确，Self-Evolve 可能固化错误模式，形成**累积误差**（越进化越错）。  
- **分布偏移**：子智能体基于历史任务优化，在工具 API 版本更迭或外部环境变化后，已存代码静默失效，且 Meta-Agent 可能无法感知。  
- **对抗性任务输入**：若攻击者注入精心构造的任务，可能诱导 Meta-Agent 将恶意逻辑持久化为子智能体代码（能力积累变成有害能力积累）。

#### 4.3 值得深挖的研究点

代码层的「有害能力积累」安全问题：现有框架对子智能体代码的保存**缺乏意图级审查**，若输入任务被污染或存在提示注入，Meta-Agent 可能将有害行为固化为可在其他系统复用的模块。研究方向：设计面向代码库的动态污点追踪与意图一致性校验机制，构成一篇结合 Agent 安全与代码分析的工作。

### 5. Motivation（思想逆向推演）

- [现有 agent 每次任务都要重新理解和规划...] -> [那么核心痛点是不是：知识/能力无法在任务间「固化」地传递？] -> [文本反思已经被用了，但文本的本质是「描述」而非「执行」，描述和执行之间永远有 gap...] -> [软件工程中函数库/工具库解决的正是「把解法固化为可重复执行的程序」这件事，能不能把这套思路搬到 agent 自进化里？] -> [如果把每次成功执行的轨迹直接编译成代码模块保存，下次同类任务直接调用，异常时再修改代码，就实现了「可执行经验积累」] -> [最终推导出三阶段生命周期 Install → Self-Evolve → Deploy 的 AgentFactory 框架]
